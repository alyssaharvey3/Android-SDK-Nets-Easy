    Terminology

    • MiA SDK - The SDK. From your reference, an Android Library.

    • Easy Checkout - Is a comprehensive online payment service provided by Nets. It enables merchants to offer their customers different ways to pay purchases easily and securely in webshops and mobile environments. When using EASY, it is easy to pay with a broad range of payment methods and currencies, and still meet high security requirements set for online payments.

    • Easy offers multiple checkout pages options – see more here

      • Full Easy checkout :

        The Full Easy checkout is the perfect solution for merchants that sell physical goods and do not offer customers to create a profile on their web shop.

      • The Payment method on File Checkout :

        The payment method on file checkout is the perfect solution for merchants that sell physical goods (Everything that goes in a box) and customers to create a profile on their web shop.

      • The Digital goods / services Checkout :

        The Digital goods / services checkout is the perfect solution for merchants that sell digital goods or services and enables customers to create a profile on their web shop.

      Note that Easy offers options of either an embedded checkout or a hosted pay page checkout (re-direct). For the SDK, the re-direct hosted page is the recommended choice. You may not be able to process Nordic wallet payments with the embedded checkout option.

    Introduction

    The Easy checkout provides a one-click checkout that handles payment methods, consumers shipping information, and stores this in a safe and secure manner. We have optimized the checkout to convert visitors to consumers by reducing the actions a potential buyer must do before completing a purchase.
    The checkout offers everything through one integration – get access to payment methods, consumers information and their stored payment methods through one fast integration.

    Getting started

    An integration towards Nets Easy requires having the paymentId(transactionID) created before the checkout is initialized. Your mobile application can use one of the available integration types, based on your business requirements:

    1. Easy Hosted Payment Window (recommended): the Easy checkout page is hosted by DIBS Easy and will handle all checkout initialization. This is the recommended approach..
    2. Merchant Hosted Payment Window (not recommended) : The checkout page will be hosted by you, and also all the checkout initialization and JavaScript callback handling. This approach is not recommended as it can cause issues when paying with mobile wallets such as Swish, Vipps or MobilePay.

    Your application must follow the next flow:

    • Create payment id
      • Your mobile application will call an API on your backend to generate the PaymentID. Your backend will call the EASY REST payment API ( DIBSPayment API) to fetch the required ID, store it in the Database, then return it back to calling application. Based on your chosen integration type, you need to consider the following:

      1. HostedPaymentPage (recommended):

        • In the request body, under checkout object, make sure to send the parameters: integrationType = HostedPaymentPage and returnURL = YOUR RETURN URL. Make sure that the parameter url is not sent. Easy will initialize the checkout, and will return to you the paymentId and the hostedPaymentPageUrl.
          Note: The value for returnUrl must be a string in URL pattern. It's not required to be a valid web page, since it won't be loaded in the WebView. It is just for detection purpose only for the SDK to know when the checkout process has completed.

      2. EmbeddedCheckout (not recommended) :

        • In the request body, under checkout object, make sure to send url = YOUR HOSTED CHECKOUT PAGE and the parameters: integrationType and returnURL should NOT be sent. Easy will return the paymentId and you will have to initialize the checkout.
        • Initialize the checkout

        The checkout HTML page will be simplified for mobile (According to Easy Best-Practice rules, the DIBS iFrame will be embedded inside your checkout page. For mobile, this page should be smaller, considering devices screen size. For example, you can build this page with the DIBS iFrame only, which will use the created paymentId and your secret checkout key).

        Required contents of the Checkout HTML page:

        • Include the checkout JavaScript URL in the page <head> tag

          //TEST environment script
https://test.checkout.dibspayment.eu/v1/checkout.js?v=1
//PROD environment script
https://checkout.dibspayment.eu/v1/checkout.js?v=1

        • Add an empty <div> tag with a custom ID (e.g. dibs-complete-checkout)

          <div id="dibs-complete-checkout"></div>

        • Introduce a page variable in a <script> tag to build all info required for the payment. The script will run once the page is loaded, and the variable will be used by the JS file.

          var checkoutOptions = {
                    //your secret Checkout key. You should retrieve this from your secure DB
                    checkoutKey: "CHECKOUT_KEY",
                    //the paymentID. You should retrieve this from your secure DB
                    paymentId : "PAYMENT_ID_KEY",
                    //user preffered language
                    language: "en-GB",
                    //the <div> element id defined at previous step
                    containerId : "dibs-complete-checkout",
                };

        • Instantiate the Dibs.Checkout object and add listener for payment completed event:

          var dibsCheckout = new Dibs.Checkout(checkoutOptions);
dibsCheckout.on('payment-completed', function(response) {
         //this line will notify the SDK that the payment process is completed and successful
         window.JSCallbackInterceptor.notifyPaymentCompleted();
});

        Important: it's mandatory that your Checkout HTML page will add the listener for 'payment-completed' event, and inside the callback function to call for window.JSCallbackInterceptor.notifyPaymentCompleted();.
        Without this implementation on your page, the SDK cannot know when the payment process has completed and when to return the user back to your application with a MiAResult object.

  • Handle Consumer Data:
    Apart from the standard checkout, EASY supports prefilling the consumer data while creating the payment ID to offer a faster checkout and better user experience. Based on your requirement, EASY supports the following types of Consumer Data Handling by the merchant.

      • Inject address mode :

        The merchant’s app can prefill the consumer data and send it to EASY Checkout while creating the payment ID. The checkout object under the request body must contain the consumer data and merchantHandlesConsumerData = true. Here is an example :

        "checkout": {
    ...
    "consumer":{
      "email":"john.doe@doemail.com",
      "shippingAddress":{
         "addressLine1":"Toodels street 1",
         "addressLine2":"Co/ John",
         "postalCode":"0956",
         "city":"Stockholm",
         "country":"SWE"
      },
      "phoneNumber":{
         "prefix":"+46",
         "number":"0123456789"
      },
      "privatePerson":{
         "firstName":"john",
         "lastName":"Doe"
      }
   },
    "merchantHandlesConsumerData": true
  }

      • No Shipping mode :

        If the purchase does not require physical shipping, then the merchant app can prefill only the email and postalCode along with merchantHandlesConsumerData = true while creating the payment ID. Here is an example :

        "checkout": {
    ...
    "consumer":{
      "email":"john.doe@doemail.com",
      "shippingAddress":{
         "postalCode":"0956",
      }
   },
    "merchantHandlesConsumerData": true
  }
        • Normal checkout Merchant handled consumer data checkout

    • Handle any modifications in the cart or shipping
    • Launch SDK

    When user clicks on Buy button from your cart page, launch the SDK. Based on your chosen integration type, you need to build the MiaPaymentInfo object:

    1. HostedPaymentPage::

    Important: Make sure the returnUrl is the same one you are sending in the API call to retrieve the paymentId. Without this, the SDK cannot detect when the payment has completed in order to deliver a result back to your application.

     MiASDK.getInstance().startSDK(this, MiAPaymentInfo(paymentId, checkoutUrl, returnUrl))
    1. EmbeddedCheckout (not recommended):
     MiASDK.getInstance().startSDK(this, MiAPaymentInfo(paymentId, checkoutUrl))

    The SDK will load the given checkoutUrl in a WebView and the user can proceed with the payment.

    • Handle Payments after Checkout

    For mobile case, a result will be delivered back to the application in onActivityResult() method, specifying the checkout status.

     override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?) {
        if (requestCode == MiASDK.EASY_SDK_REQUEST_CODE) {

            if (resultCode == Activity.RESULT_OK) {
                val result = data?.getParcelableExtra<MiAResult>(MiASDK.BUNDLE_COMPLETE_RESULT)

                when (result?.miaResultCode) {
                    //user completed the payment
                    MiAResultCode.RESULT_PAYMENT_COMPLETED -> {
                       //call getPayment()
                    }
                    //user has cancelled the payment
                    MiAResultCode.RESULT_PAYMENT_CANCELLED -> {
                       //call getPayment()
                    }
                    //user encountered and error and cannot proceed with the payment
                    MiAResultCode.RESULT_PAYMENT_FAILED -> {
                        //the SDK has an internal error and cannot proceed with the current payment
                    }
                }
            }

            return
        }
        super.onActivityResult(requestCode, resultCode, data)
    }

    Note: The SDK result does not represent the payment status. In order to confirm if the payment was successful or not, you need to call Get Payment and make verifications on the summary object:

      "summary": {
    "cancelledAmount": 0,
    "chargedAmount": 0,
    "refundedAmount": 0,
    "reservedAmount": 0
  }

    Important: Once a purchase is completed, the customer must receive a confirmation of the order and a receipt in the form of an on-screen-display and e-mail. Meaning that after you call Charge Payment, you already have the payment information retrieved at previous step, and display it to the user.

    Subscriptions (Recurring API)

    EASY supports subscriptions allowing charging customers on a regular basis. The recurring API contains subscription creation and charging of subscriptions. Merchant is responsible for managing subscriptions after they're created with EASY recurring API. The following code samples demonstrate the steps required to create and charge subscriptions.

    Create a Subscription with EASY Recurring API

    To create a subscription, include a subscription JSON object in the request body of the POST payment.

    See API documentation

     
// Subscription request body
data class RegisterPaymentRequest {
	val order: Order
	val checkout: Checkout
	val subscription: Subscription
	val merchantNumber: String?
	val notifications: String?
}
data class Subscription {
	val endDate: String 
	val interval: Int 
}
	
// Response object following subscription creation
data class RegisterPaymentResponse {
	val paymentId: String 
	val hostedPaymentPageUrl: String 
}	
	
// E.g. subscription creation request. See MiaSample app for more	
@POST("v1/payments")
fun createSubscription(
	@Body(body: RegisterPaymentRequest 
): Call<RegisterPaymentResponse>

    A subscription object contains an endDate, last date of the subscription, and an interval representing the interval (in number of days) at which the subscription can be charged. For recurring payments chargeable at any given time (with no regular interval), interval = 0 should be used.

    EASY recurring API expects the endDate to have the following format: yyyy-MM-dd'T'HH:mm:ss+zz:zz.

    e.g. 2019-07-18T00:00:00+01:00 for timezone GMT + 1 hour

    Retrieve a Newly Created Subscription

    EASY recurring API returns paymentID following subscription creation. To obtain the subscription details, use this ID to fetch subscription ID for the newly created subscription.

    See API documentation

     
// Response object following fetch request of subscription with `paymentId`
// Object contains subscription ID that can be used to fetch subscription details
data class SubscriptionRegistrationResponse {
	val payment: Payment 
}	
data class Payment {
	val created: String 
	val subscription: Subscription 
}		
data class Subscription {
	val id: String 
}		
	
// E.g. payment (for subscription) request. See MiaSample app for more	
@POST("v1/payments/{PAYMENT_ID}")
fun fetchSubscriptionPayment(
	@Path("PAYMENT_ID") paymentId: String 
): Call<SubscriptionRegistrationResponse>

    Subscription details can be obtained from EASY using a subscription ID.

     
// Subscription Details - Response to subscriptions fetch for given `subscriptionID`	
data class SubscriptionDetailsResponse {
	val subscriptionId: String 
	val interval: String 
	val endDate: String 
	val paymentDetails: PaymentDetails 
}	

//  E.g. subscription details fetch request. See MiaSample app for more	
@POST("v1/subscriptions/{SUBSCRIPTIONID}")
fun fetchSubscriptionDetails(
	@Path("SUBSCRIPTIONID") subscriptionId: String 
): Call<SubscriptionDetailsResponse>

    Charging Subscription

    Subscriptions can be charged via a POST charge request providing subscription ID and a charge request body containing order details.

    See API documentation

     
// Charge Subscription request body
data class ChargeRequest {
		val order: Order
}

// Success response following charge-subscription request
data class ChargePaymentResponse {
	val chargeId: String 
	val paymentId: String 
}	
	
// E.g. charge request. See MiaSample app for more	
@POST("v1/subscriptions/{SUBSCRIPTIONID}/charges")
fun chargeSubscription(
	@Path("SUBSCRIPTIONID") subscriptionId: String 
	@Body(body: ChargeRequest
): Call<ChargePaymentResponse>

    After creating a subscription, merchant is responsible for storing and managing the subscription.

    (Note that the MiaSample app stores subscriptions in the device for demo purposes but we recommend your app backend for storage)

    Error codes handling

    The SDK will deliver the result in your calling activity/fragment's onActivityResult() under MiAResult object.

    Based on the content of this object, you must take the following actions:

    • MiAResultCode: enum item, and can have one of the three codes:

      • RESULT_PAYMENT_COMPLETED: the checkout process has completed; to determine the payment status, perform getPayment API call.
      • RESULT_PAYMENT_CANCELLED: the checkout process has been cancelled by the user; you still need to perform the getPayment API call to find the payment status
      • RESULT_PAYMENT_FAILED: the SDK has encountered an internal error; the checkout process is unable to load (the payment has not yet been started). You need to show a custom error to the user

    • miaError: it’s an enum item. It will be a non-null value only if miaResultCode == MiAResultCode.RESULT_PAYMENT_FAILED. This enum may have the following values:

      • MiASDKError (errorCode = 101): this occurs when something goes wrong inside the SDK (e.g. values for paymentId or checkoutUrl are lost)

    Note: For notifying the user about this error, we recommend that you map your own error message based on this errorCode. This way you can show your customers the message which suits you better.
